# Architecture: How Socotra Plugin Java Fits Together

Purpose: explain the two-layer Java model (stable core-datamodel JAR vs per-configuration generated `com.socotra.deployment.customer` package), where plugin implementations live, and how typed request/response dispatch works. Examples use the single-product "ZenCover" config (`generated-examples/zencover/`, core-datamodel v1.7.61); a multi-product contrast lives in `generated-examples/credit-card-protection/` (v1.6.180).

## Two layers of code

### 1. core-datamodel JAR (stable platform API, version-pinned, e.g. v1.7.61)

Hand-written + codegen'd platform classes. Identical across all tenants/products. Packages:

| Package | Contents |
|---|---|
| `com.socotra.coremodel` | ~300 platform records/enums/interfaces: `Policy`, `Transaction`, `Term`, `Charge`, `RatingSet`, `RatingItem`, `ValidationItem`, `UnderwritingFlag(s)`, `Installment`, `Invoice`, plugin response records (`CancellationPluginResponse`, `PreCommitTransactionResponse`, ...), enums (`QuoteState`, `TransactionState`, `DurationBasis`, ...), `@Plugin`/`@AutomationPlugin` annotations, `PluginType`, `PluginRequest`, `TableMetadata`/`RangeTableMetadata`/`ConstraintTableMetadata`, `Chargeable`, `CustomerObject` |
| `com.socotra.coremodel.interfaces` | Product-agnostic views of generated types: `Quote`, `QuoteCore`, `Segment`, `Account`, `Policy`, `Contact`, `Fnol`, `Producer*`, `Payment`, `Element`, `SelectableResource`, `ValidationResult` |
| `com.socotra.coremodel.constraints` | Field constraint annotations/handling: `Required`, `MinValue`, `MaxValue`, `MinLength`, `MaxLength`, `Options`, `Regex`, `StaticConstraint` |
| `com.socotra.coremodel.views` | Jackson `@JsonView` markers: `Internal`, `Public` |
| `com.socotra.deployment` | Runtime services: `DataFetcher` (+ `DataFetcherFactory`), `ResourceSelector` (+ factory), `TableRecordFetcher`, `RangeTableRecordFetcher`, `ConstraintsFetcher`, `MoneyService`, `TimeService`, `EventsService`, `AuxDataService`, `SearchService`, `DeploymentFactory`, `DeploymentConfig`, `StreamingEntity` |
| `com.socotra.deployment.plugins` | `ConfigMigrationTransformer`, `PluginExecutionContext`, `AutomationPluginException` |
| `com.socotra.deployment.schedules` | `RatingRegistry`, `Schedule` (rating schedules) |
| `com.socotra.deployment.workmanagement` | `WorkManagementService`, `TasksConfig` |
| `com.socotra.deployment.producermanagement` | `ProducerManagementConfig` |
| `com.socotra.developer` | `APIClient`, `APIDataFetcher`, `APIResourceSelector` (local dev/testing against the REST API) |
| `com.socotra.platform.tools` | `ULID`, `TableUtils`, `NumberUtils`, `Interpolation` |

### 2. Generated `com.socotra.deployment.customer` package (per-configuration, regenerated on every config deploy)

Generated from the deployed configuration. One config may define **multiple products** (e.g. the credit-card-protection example generates a single customer package covering both `BasicCreditCardProtection` and `PremiumCreditCardProtection`). Contains:

- **Plugin SPI interfaces** — `RatePlugin`, `ValidationPlugin`, `UnderwritingPlugin`, `PreCommitPlugin`, `InstallmentsPlugin`, `CancellationPlugin`, `RenewalPlugin`, `DelinquencyEventPlugin`, `DocumentSelectionPlugin`, `DocumentDataSnapshotPlugin`, `DocumentConsolidation{Selection,Snapshot}Plugin`, `ConfigMigrationPlugin`, `AutopayPlugin`, `PaymentPostProcessingPlugin`, `Workplan{Selection,Execution}Plugin`, plus `@AutomationPlugin`-annotated interfaces. Interface *names* are stable; their nested request record names and overloads embed the **product name** (e.g. `ZenCoverQuoteRequest` for a product named "ZenCover"), with one overload set per product in multi-product configs. Which interfaces are generated at all depends on config features (payment methods, automations) and on the core-datamodel version the config targets — see `references/plugin-interfaces.md` § "What varies per configuration".
- **Typed product model records** — quote/segment/policy shapes per element: `ZenCoverQuote`, `ZenCoverSegment`, `ZenCoverPolicy`-style records; per-exposure/peril `Item`/`ItemQuote`/`ItemPolicy`, `Theft`/`TheftQuote`/`TheftPolicy`, etc.; `*Data` records for config-defined fields; account types (`PersonalAccount`, `BankCustomerAccount`, ... — one type per configured account, names follow config); enums `ChargeType` and `TransactionType` implementing the coremodel interfaces of the same names; table records (`CountyRiskFactors implements CustomerObject, TableMetadata`); plan configs (`InstallmentPlans`, `DelinquencyPlans`, ...); `CustomerDeployment implements DeploymentFactory`.

These files are headed `Autogenerated - DO NOT EDIT!`. Plugin authors compile **against** them but never modify them.

## Where plugin implementations live

Implementation classes are part of the product config, under `plugins/java/` (deployed bundle path: `.../config/.../plugins/java/*.java`). They are compiled by the platform into `customer-plugins.jar` (the generated model goes into `customer-config.jar` — constants on `com.socotra.deployment.DeploymentFactory`). An implementation is a plain class in package `com.socotra.deployment.customer` implementing a generated interface:

```java
package com.socotra.deployment.customer;

import com.socotra.coremodel.RatingItem;
import com.socotra.coremodel.RatingSet;

public class ZenCoverRatingPlugin implements RatePlugin {
    @Override
    public RatingSet rate(ZenCoverQuoteRequest request) {
        ZenCoverQuote quote = request.quote();
        ...
        return RatingSet.builder().ok(true).ratingItems(ratingItems).build();
    }
}
```

No registration code is needed: the platform discovers the class via the `@Plugin` annotation on the interface it implements.

## The @Plugin and @AutomationPlugin annotations

From `com.socotra.coremodel` (source-verified):

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface Plugin {
  PluginType type();
}

@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface AutomationPlugin {
  String type();   // free-form automation name, e.g. @AutomationPlugin(type = "CancellationInvoices")
}
```

The annotations are placed on the **generated interfaces** (e.g. `@Plugin(type = PluginType.rating) public interface RatePlugin`), not on your implementation class. Your class is associated by `implements`.

`PluginType` (enum in coremodel) is the dispatch registry. Each constant carries the generated interface FQCN, a stub class name, a method name, and a stateless method name. The table below is the **v1.7.61** registry; older datamodels carry fewer constants — v1.6.180 has only 11 (`rating, underwriting, validation, documentSelection, documentDataSnapshot, preCommit, renewal, delinquencyEvent, documentConsolidationSnapshot, installments, autopay`; verified via `javap`), so the missing interfaces (`CancellationPlugin`, `ConfigMigrationPlugin`, `PaymentPostProcessingPlugin`, `DocumentConsolidationSelectionPlugin`, workplan, automation, deserialization) simply do not exist in packages generated against it:

Each generated interface also nests a no-op `<Name>PluginStub` (`PluginType.getStubName()` returns `className + "$" + stubName`) used when no customer implementation exists, and declares `public static final Integer VERSION = n;`.

## Typed request/response flow

1. A lifecycle event fires (e.g. quote priced, transaction issued, document generation).
2. The platform builds a **request record** — a nested record inside the generated interface, implementing `com.socotra.coremodel.PluginRequest` (`Collection<?> structures();`). The records have static `of(...)` factories that load typed objects via `DataFetcherFactory.get()` — e.g. `RatePlugin.ZenCoverQuoteRequest.of(quoteLocator, duration, durationBasis)` calls `getQuote(...)` which returns the *generated* `ZenCoverQuote`.
3. Dispatch selects the implementation's overload **by request record type**: quote-level events call `rate(ZenCoverQuoteRequest)`, segment/transaction-level events call `rate(ZenCoverRequest)`, quick-quotes call `rate(ZenCoverQuickQuoteRequest)` — and in a multi-product config the request type also selects the product (e.g. `rate(BasicCreditCardProtectionQuoteRequest)` vs `rate(PremiumCreditCardProtectionQuoteRequest)`). All interface methods are `default` with safe no-op bodies, so an implementation overrides only what it needs.
4. The return value is a coremodel record (`RatingSet`, `ValidationItem`, `UnderwritingModification`, ...) consumed by the platform. Returned records are validated in their canonical constructors (e.g. `RatingItem` requires `rate` for `ChargeHandling.normal` and `amount` for `flat`/`retention`).

`stateless*` variants default to delegating to the stateful method (`default RatingSet statelessRate(ZenCoverQuoteRequest r) { return rate(r); }`); override them only when stateless (no side-effecting fetches) execution must differ.

Gotcha: request `of(...)` factories for transaction-scoped requests throw `IllegalStateException` if the transaction is still in `TransactionState.draft`.

## Key compile-time implication

Plugin code compiles against **both** jars. Anything config-specific (`ZenCoverQuote`, `ItemQuote`, `ChargeType.premium`, request record names) comes from the generated package and changes when the config changes; anything in `com.socotra.coremodel`/`com.socotra.deployment`/`com.socotra.platform.tools` is the stable contract. Generic methods such as `DataFetcher.<T> T getQuote(ULID)` are typed at the call site by the generated product types.
